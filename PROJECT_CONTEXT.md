# WebcamExpert MVP: контекст проекта

Последнее обновление: 2026-06-17, MSK.

Этот файл нужен как рабочая память проекта. Перед новыми улучшениями, правками визуала, SEO, ролей, сценариев или функций нужно сначала сверяться с ним, чтобы не переделывать уже реализованное и не возвращаться к старым решениям по кругу.

## Что это за проект

WebcamExpert MVP — UGC-медиа и комьюнити-платформа для вебкам-индустрии.

Ориентиры по продукту и интерфейсу:
- По механике ближе к vc.ru: лента, авторы, реакции, комментарии, обсуждения, профили, вакансии/услуги.
- По подаче ближе к Т-Ж: рубрики, редакционные подборки, понятная типографика, SEO-посадочные страницы, практичные разборы.
- Это не каталог студий. Главная ценность — пользовательские истории, разборы, опыт авторов, полезные материалы, вакансии, услуги и резюме.

Основная продуктовая логика:
- Пользователь может быть потребителем, поставщиком услуг или совмещать оба режима.
- Обычная модель может читать, писать статьи как личный блог, комментировать, размещать резюме, искать вакансии/услуги.
- Поставщик может публиковать экспертные статьи, вакансии и услуги.
- Сейчас публикации бесплатные.

## Важные принятые решения

- Внутренние ссылки на статьи используются через `id`: `/articles/{id}`. Это надежнее для текущего MVP, потому что старые кириллические slug могли давать 404.
- Внутренние ссылки на конкретные вакансии/услуги используются через `id`: `/listings/{id}`.
- В ленте статей карточка должна быть простой:
  - автор виден только сверху;
  - нижняя строка `Автор:` убрана;
  - отдельная кнопка `Репост` в карточке убрана;
  - метрики сведены в одну строку: просмотры, реакции, комментарии, репосты;
  - действия в карточке: подписаться на рубрику, пожаловаться.
- Репост оставлен на странице конкретной статьи, а не в карточке ленты.
- На странице статьи репост не должен быть серверной кнопкой-счетчиком. Он должен отдавать пользователю ссылку на статью и копировать ее в буфер.
- Кнопка `Обсудить` на странице статьи не должна увеличивать отдельный счетчик. Она ведет к блоку комментариев.
- Подписки на автора и рубрику работают как toggle: после подписки кнопка должна превращаться в `Отписаться`, после отписки — обратно в `Подписаться`.
- Жалоба должна возвращать пользователя на исходную страницу с видимым подтверждением отправки.
- При написании статьи автор обязан выбрать рубрику. Сервер тоже проверяет рубрику, не только UI.
- Платный след в пользовательском интерфейсе убран. Платежные модели/таблицы могут оставаться как будущий скрытый модуль, но не должны светиться пользователю как текущая механика.

### Stable-сервер и ассеты

- `scripts/dev-stable.sh` после build запускает `scripts/ensure-webpack-runtime.mjs`.
- Это восстанавливает `.next/server/webpack-runtime.js`, если Next 14 в текущем окружении не создал его сам. Без этого auth/API route может падать с `Cannot find module '../../../../webpack-runtime.js'`.
- `public/legacy-next-static.js` усилен для старых chunk-запросов: регистрирует ожидаемые chunk ids и переводит вкладку на свежий URL с новым `assetRefresh`.

### Резюме

- Пользователь может бесплатно разместить резюме.
- Резюме публикуется на 7 дней. После окончания срока оно скрывается из публичного каталога, но остается в кабинете автора и может быть продлено кнопкой `Продлить на 7 дней`.
- В кабинете резюме модели заполняется как quiz-анкета, а не как одно большое текстовое поле.
- Визуально это квиз:
  - один фокусный блок на экране;
  - верхний прогресс `Шаг X из 7`;
  - карточки-ответы вместо обычных селектов;
  - кнопки `Назад` / `Далее`;
  - мини-превью блока “о себе”.
- Шаги quiz:
  - основное;
  - о себе;
  - график;
  - деньги;
  - требования к студии;
  - стоп-лист;
  - контакты.
- Анкета модели собирается из селектов, чекбоксов и коротких полей:
  - возраст;
  - город для каталога;
  - часовой пояс / формат;
  - опыт;
  - языки;
  - смены в неделю/месяц;
  - длительность смены;
  - пропуски;
  - минимальный процент;
  - минимальный доход;
  - частота выплат;
  - оборудование;
  - условия комнаты;
  - интерьер;
  - организационные условия;
  - стоп-студии;
  - запрещенные условия.
- Серверная ветка `resumeTemplate=model-v1` собирает из этих полей структурированный `bio` по блокам. Это нужно, чтобы студия быстро понимала требования модели.
- Город хранится отдельно в поле `city` как значение каталога (`Москва`, `Санкт-Петербург`, `Казань`, `Екатеринбург`, `Минск`, `Удаленно`). Часовой пояс не нужно склеивать с городом, иначе резюме не попадет в городскую выдачу `/resumes`.

## Роли и сценарии

### Гость

- Читает главную, статьи, авторов, вакансии, услуги, резюме, SEO-страницы.
- Не может открыть кабинет без входа.
- Может зарегистрироваться.

### Модель / обычный пользователь

- Пишет личную статью.
- Выбирает рубрику статьи.
- Размещает резюме.
- Читает статьи.
- Ставит реакции.
- Комментирует.
- Отвечает на комментарии.
- Лайкает комментарии.
- Репостит на странице статьи.
- Откликается на вакансии.
- Сохраняет вакансии/услуги.

### Студия / поставщик

- Публикует экспертную статью.
- Публикует вакансию.
- Публикует услугу.
- Видит контакты резюме после входа.
- Не имеет доступа в админку.

### Эксперт

- Публикует экспертную статью.
- Публикует услугу.
- Не имеет доступа в админку.

### Админ

- Видит админку.
- Работает с жалобами.
- Может скрыть или разрешить жалобу.

## Что не считать багом

- Платежные таблицы и старые payment actions могут оставаться в коде как будущий скрытый модуль, если они не видны пользователю.
- Репост в ленте специально убран. Репост должен быть на странице статьи.
- Репост на странице статьи — это выдача ссылки/копирование ссылки, а не серверное увеличение счетчика.
- `Обсудить` — это переход к комментариям, а не отдельная реакция-счетчик.
- Автор в карточке ленты должен быть только сверху.
- Рубрика статьи должна выбираться автором явно.
- Автоподбор рубрики сервером допустим только для служебных/старых данных и черновиков, но публикация новой статьи должна требовать рубрику.

## Как работать дальше

Перед каждой новой правкой:

1. Открыть этот файл.
2. Проверить, не реализована ли уже функция.
3. Если меняется принятое решение, явно обновить этот файл.
4. После правки добавить сюда краткую запись:
   - что изменено;
   - где изменено;
   - какие проверки пройдены;
   - есть ли новые ограничения или решения.

Минимальный набор проверок после UI/функциональных изменений:

- `npm run build`
- `npm run qa:roles`
- `npm run check:links`
- браузерная проверка затронутых страниц.

Если после production build нужно вернуть сайт в браузер:

1. Остановить процессы на `:3000`.
2. Очистить `.next`.
3. Запустить `npm run dev:stable`.

`npm run dev:stable` специально запускает production-like режим: чистит `.next`, делает `npm run build`, проверяет наличие `.next/static` и запускает `next start`. Это нужно для визуальной проверки сайта, потому что обычный `next dev` в этой локальной среде может отдавать HTML без `_next/static` CSS/JS ассетов. Визуально это выглядит как “слетел весь дизайн”, особенно после полной навигации или `Выйти`.

`npm run dev` теперь тоже запускает стабильный production-like режим. Сырой Next dev оставлен как `npm run dev:raw`, но его не использовать для пользовательского просмотра.

Дополнительные защиты от “голого HTML” и `Application error` после старых dev-чанков:
- `src/auth.ts` содержит `trustHost: true`, чтобы Auth.js доверял `localhost:3000`.
- Security headers в `next.config.mjs` больше не вешаются на `/_next/static`, иначе production static мог получать 400.
- Обычные HTML/RSC-страницы получают `Cache-Control: no-store, must-revalidate`, чтобы браузер не держал старый HTML со старыми chunk-путями.
- `src/middleware.ts` переписывает старые dev chunk URL (`/_next/static/chunks/app/page.js`, `webpack.js`, `main-app.js`, `app-pages-internals.js`, старый `layout.css`) на legacy-файлы из `public/`, которые форсят свежую загрузку страницы или дают пустой CSS-заглушку.
- `scripts/dev-stable.sh` после production build копирует текущий hashed CSS в `public/legacy-next-static.css`. Поэтому даже если Safari держит старый HTML с dev-путем `/_next/static/css/app/layout.css`, он получает настоящий Tailwind CSS, а не пустую заглушку.

Не запускать `npm run build` параллельно с `next dev` или браузерным обходом. Для показа сайта пользователю и финальной визуальной QA использовать `npm run dev`.
